   identified in the post-merge audit maps 1:1 to exactly one acceptance
   criterion (AC1-AC4). No gap is orphaned; no AC is untethered.

2. **Hard-gate phrasing.** All four ACs use deterministic-failure language
   ("fail deterministically", "must trigger", "must prove") rather than
   advisory or documentation-only language. The material deviation rules
   explicitly prohibit weakening these to warnings. This directly answers the
   caller's question: the acceptance criteria are sufficient to prevent a
   false-fixed claim.

3. **Scope boundary is correctly drawn.** The lane stays in
   governance-source only; downstream consumer repos are out of scope.
   Reopening #583 is excluded unless required by residual findings.

5. **Sprint structure is sound.** Sprint 1 (packet + review) is cleanly

### Minor findings (non-blocking, address before or during implementation)

**M1. AC1 should name the concrete gate mechanism.**
AC1 says "any branch containing an issue token must fail the mandatory-plan
gate" but does not specify that the fix is to drive the gate from the existing
`_branch_issue_number()` helper rather than adding new prefix matching. The
validator-gap specialist already identified this. Recommend adding a
one-sentence implementation note to AC1 referencing
`_branch_issue_number(branch)` as the intended gate function, so the
implementation worker does not re-invent prefix matching.

**M2. AC2 should specify the minimum schema fields.**
AC2 requires "alternate-family or alternate-identity reviewer independence"
but does not name the concrete schema additions (`reviewer_model_id`,
`reviewer_model_family`) that the validator-gap specialist recommended.
Recommend pinning these field names in the AC or in a linked schema-change
spec so the implementation scope is unambiguous.

**M3. Backward compatibility of schema changes is unaddressed.**
Adding required fields to `alternate_model_review` (per M2) will
retroactively invalidate existing valid packets on other branches unless the
validator version-gates or the schema uses conditional requirements. The PDCAR
should note whether this is intentional (all existing packets must be
re-validated) or whether the new fields are only enforced when
`alternate_model_review.status == "accepted"`.

**M4. Validation artifact should include clean revalidation proof.**
The validation artifact documents initial failures and states the packet "can
be revalidated cleanly" but does not include the actual clean-run output.
Before promotion, rerun the validation commands and append the passing output.

### Risks (not findings, for awareness)

- **R1.** AC4's session-contract semantic validation ("prove the canonical
  bootstrap/runbook helper contract more concretely than file presence alone")
  is the least precisely specified AC. The session/runbook specialist's
  recommendation (check for canonical bootstrap command string,
  `--emit-hook-note` helper contract, wrapper->helper path) provides concrete
  targets, but the AC itself leaves room for an implementation that only
  marginally improves on presence checks. Recommend the implementation sprint
  acceptance criteria reference the specialist's three specific semantic
  checks.

- **R2.** The structured plan lists `docs/EXTERNAL_SERVICES_RUNBOOK.md` as a
  research artifact and an AC4 verification ref but does not list it in Sprint
  2's `file_paths`. If the runbook needs patching to stay aligned with the new
  machine checks (as the PDCAR Do section suggests), it should appear in the
  sprint file manifest.
